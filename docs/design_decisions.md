# 設計上の判断メモ

課題提出用のため、汎用性よりシンプルさを優先した箇所をまとめます。

---

## 画像添付の管理方法

**採用した設計：** `messages`テーブルに`image_path`カラムを追加

**汎用的な設計：** `attachments`テーブルを別途作成し、`message_id`で紐づける

**理由：**
- 複数枚の画像を一緒に表示する要件はないため
- 中間テーブルを使うとJOINが増えて複雑になる

---

## トークルームの参加者管理

**採用した設計：** `rooms`テーブルに`created_user_id`と`invited_user_id`の2カラム

**汎用的な設計：** `room_members`中間テーブルで参加者を管理する

**理由：**
- 要件が「1:1メッセージ」のみのためグループチャット対応が不要
- 中間テーブルを使うとJOINが増えて複雑になる

---

## ユーザー認証

**採用した設計：** `username`と`password`のみ

**汎用的な設計：** `email`も持ち、パスワードリセット機能を実装する

**理由：**
- 要件にメールアドレスの記載がないため省略
- パスワードリセットは要件外

---

## ディレクトリ構造（共通パーツの配置）

**採用した設計：**
```
public/
├── partials/
│   └── sidebar.php   # 共通パーツ
├── top.php
├── user_list.php
└── auth/
src/
└── Model/            # DBアクセスのみ
```

**汎用的な設計：** `src/View/`にテンプレートをまとめ、`public/`はリクエストの入口のみにするMVC的な構成

**理由：**
- フレームワークなしのシンプルなPHPアプリのため、`public/`にページファイルをまとめる方が直感的でわかりやすい
- `src/View/`に共通パーツだけ置く中途半端な状態より、`public/partials/`にまとめた方がファイルの役割が明確になる
- `src/`はDBアクセス（Model）に専念させることで責務がシンプルになる

## リアルタイムメッセージ受信

**採用した設計：** ポーリング（JavaScriptが2秒ごとにサーバーへ新着メッセージを問い合わせる）

**汎用的な設計：** WebSocketを使いサーバーからブラウザへ即時プッシュ配信する

**理由：**
- WebSocketはPHPと相性が悪く、Ratchetなどの専用ライブラリや常駐プロセスが必要
- 提出先のレンタルサーバー（xbiz.jp）がWebSocketに対応していない可能性が高い
- 要件は「ページ更新なしで受信できること」であり、ポーリングで十分満たせる
- ポーリングはAjaxで定期的にAPIを叩くだけなのでシンプルに実装できる

**ポーリングのメリット・デメリット：**

| | 内容 |
|--|------|
| メリット | 実装がシンプル・どんな環境でも動く・デバッグしやすい |
| デメリット | 最大2秒の遅延・新着なしでもリクエストが発生・ユーザー増加で負荷が増大 |

本番サービスでユーザー数が増えた場合はSSE（Server-Sent Events）やWebSocketへの切り替えを検討する。

## ロジックとViewの分離

**採用した設計：** `room_list.php`のように1ファイルにバックエンド処理とHTML表示を混在させる

**汎用的な設計：** DjangoのようにView（ロジック）とTemplate（表示）を別ファイルに分離する

**理由：**
- フレームワークなしのPHPでは1ファイルにまとめる書き方が一般的
- ファイル数を増やさずシンプルに保てる
- ロジックをファイル上部、HTMLを下部にまとめることで可読性を確保している
